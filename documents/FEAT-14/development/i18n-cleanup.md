# Implementation Plan: i18n / `site_content` Cleanup

## Overview

Remove i18n keys that are surfaced by the `/dashboard/content` (文案管理) admin editor but no longer have any consumer in the customer frontend. The editor enumerates **every** row in the Supabase `site_content` table, so an orphan row keeps appearing as an editable card with `預設：—` (no JSON default) — which is misleading and tempts the admin to edit text that has zero on-site effect.

Two scopes:

1. **Phase A — ship with this ticket, BEFORE the FE patch deploys.** Delete three confirmed-orphan rows whose top-level key is `story.*`. These are leftovers from a pre-`content_blocks` era when the homepage rendered "周爸的故事" as inline `t('story.p1')` paragraphs; the section was migrated to the `content_blocks` table long ago and the i18n keys were never cleaned up.
2. **Phase B — ship AFTER the FE carousel patch deploys.** Once `<HeroCarousel>` replaces the inline hero, `home.subtitle` is no longer referenced by any frontend code and can be removed from BOTH the `site_content` row and the JSON defaults. `home.title` is **kept** because `Header` and `Footer` still consume it.

## Files to Modify

### Database (Supabase MCP)

- `public.site_content` — delete three rows in Phase A, one row in Phase B.

### Frontend Source

- `shared/src/i18n/zh.json` — delete `"home.subtitle"` key in Phase B.
- `shared/src/i18n/en.json` — delete `"home.subtitle"` key in Phase B.
- `shared/dist/i18n/{zh,en}.json` — regenerated by `npm run build` from `shared`. No manual edit needed.

### Documentation

- This file already records the decision; no other docs change.

## Step-by-Step Implementation

### Phase A: Delete `story.*` orphan rows

#### Step A1: Verify the rows still exist (sanity)

**Tool:** `mcp__plugin_supabase_supabase__execute_sql`

**Query:**

```sql
SELECT key, value_zh, value_en
FROM public.site_content
WHERE key LIKE 'story.%'
ORDER BY key;
```

**Expected:** three rows returned (`story.p1`, `story.p2`, `story.title`).

**Rationale:** Catches the case where someone already cleaned them up — never assume, always check.

#### Step A2: Delete the rows

**Tool:** `mcp__plugin_supabase_supabase__execute_sql`

**Query:**

```sql
DELETE FROM public.site_content
WHERE key IN ('story.p1', 'story.p2', 'story.title');
```

**Expected:** 3 rows affected.

**Rationale:** The `WHERE key IN (...)` form is safer than `WHERE key LIKE 'story.%'` because it cannot accidentally match a future legitimate `story.*` key added by a teammate while this branch was open.

#### Step A3: Confirm the editor no longer shows the `story` tab

**Manual verification:** Open `https://papa-bread-admin.vercel.app/dashboard/content` (or the local dev URL), refresh, and confirm that the tab list at the top no longer contains a `story` entry. The `groupRowsBySection` function in `admin-frontend/src/lib/content-keys.ts:29` derives the tab list from existing rows, so an empty key prefix means an empty tab.

**Rationale:** End-to-end visual confirmation that the deletion took effect.

### Phase B: Retire `home.subtitle` (after FE carousel patch deploys)

#### Step B1: Confirm the carousel patch is live

**Manual verification:** Visit the customer storefront in production. The hero must render via `<HeroCarousel>` (no `t('home.subtitle')` call remaining in `frontend/src/app/page.tsx`).

**Rationale:** Phase B is the irreversible-on-deploy part. If Phase B happens *before* the FE patch is live, the existing inline hero falls back to an empty subtitle for one render cycle.

#### Step B2: Delete the `home.subtitle` override row

**Tool:** `mcp__plugin_supabase_supabase__execute_sql`

**Query:**

```sql
DELETE FROM public.site_content WHERE key = 'home.subtitle';
```

**Expected:** 1 row affected.

**Rationale:** No FE consumer remains. Even if a future feature wants a `home.subtitle`, it would re-add the JSON default and a fresh row.

#### Step B3: Remove `home.subtitle` from JSON defaults

**Files:** `shared/src/i18n/zh.json`, `shared/src/i18n/en.json`

**Changes:** delete the `"subtitle"` line inside the `"home": { … }` object in both files.

```diff
   "home": {
     "title": "周爸烘焙坊",
-    "subtitle": "用心烘焙，傳遞幸福",
     "allCategories": "全部",
     "editorialToggle": "介紹",
     "addToCart": "加入購物車",
     "addedToCart": "已加入購物車"
   },
```

(Same shape in `en.json`.)

**Rationale:** Removing the JSON default ensures any future stray `t('home.subtitle')` call in code becomes a missing-key warning rather than silently rendering legacy text.

#### Step B4: Rebuild shared package

**Tool:** Bash

**Command:**

```bash
cd shared && npm run build
```

**Rationale:** `@repo/shared` emits CommonJS (per CLAUDE.md). The `dist/` JSON files are compiled artifacts; rebuilding ensures both frontends pick up the change.

#### Step B5: Type-check both frontends

**Commands:**

```bash
cd frontend && npx tsc --noEmit
cd admin-frontend && npx tsc --noEmit
```

**Rationale:** Catches any leftover `t('home.subtitle')` references that would now be invalid keys (TypeScript only catches them if the i18n hook is typed against the JSON shape — this codebase is not, so the check is best-effort).

#### Step B6: Manual storefront smoke test

- Visit `/` → hero renders via the carousel; no `home.subtitle` text appears anywhere.
- Visit `/cart`, `/orders`, `/profile`, `/checkout/success` → no missing-key warnings in the browser console.

**Rationale:** Defence in depth against a TypeScript miss.

## Notes

- **Why not delete `home.subtitle` and `story.*` together in one shot?** The `home.subtitle` row is currently consumed by the live storefront. If we delete the row before the FE patch ships, customers see an empty subtitle for the duration of the gap. The two-phase ordering is purely a deployment-safety concern.

> **Review note (2026-04-28) — `useSiteContent` has a 5-minute `staleTime`:** `frontend/src/queries/use-site-content.ts:7` configures `staleTime: 5 * 60 * 1000`. Even after the Phase B `DELETE` lands, every customer with a fresh session will continue to see the old `home.subtitle` override (if any) for up to 5 minutes after their `useSiteContent` data was last fetched, until the next refetch. The plan says "no FE consumer remains" after the FE patch ships, which makes the staleness moot — but worth pinning so a reviewer doesn't worry about a multi-minute window where stale `home.subtitle` strings would still flow through `mergeOverrides`. After Phase B Step B3 (JSON default removal) the key is gone from `defaults` too, so even a stale override has nothing to overlay onto.

> **Review note (2026-04-28) — JSON default lives nested under `home: { ... }`:** `shared/src/i18n/zh.json` stores `home.subtitle` as `"home": { "subtitle": "..." }` (a nested object), not as a flat `"home.subtitle"` key. The Step B3 diff is correct in spirit — delete the `subtitle` line *inside* the `home` object — but make sure the implementer matches the existing nesting and does not accidentally introduce a top-level `"home.subtitle"` flat key (which `mergeOverrides` and `flatten()` in `admin-frontend/src/lib/content-keys.ts` would treat as a separate row entirely). `en.json` has the same nesting.

> **Review note (2026-04-28) — `groupRowsBySection` rebuild required after Phase A:** `admin-frontend/src/lib/content-keys.ts:29 groupRowsBySection` derives the editor's tab list from `site_content` rows. After deleting all three `story.*` rows, the `story` tab disappears immediately on the *next* `useSiteContent` refetch (5-min `staleTime`). For an instant verify-in-admin loop, an admin user can reload the `/dashboard/content` page or call `qc.invalidateQueries(['api','site-content'])` from devtools. Step A3 says "refresh and confirm the tab list at the top no longer contains a `story` entry" — that path works, but pin the staleTime so the reviewer expects to refresh, not wait.

> **Review note (2026-04-28) — admin-frontend has its own `useSiteContent`:** `admin-frontend/src/queries/useSiteContent.ts` exists as a separate hook for the content editor. Confirm it does not also store `home.subtitle` or `story.*` overrides in localStorage / IndexedDB (it doesn't — only TanStack Query's in-memory cache, no `persistQueryClient`). So no client-side cleanup beyond the DB delete is required.
- **The `home-content-blocks.tsx` component on the customer FE reads from the `content_blocks` table, not from i18n.** Confirm: there is no `t('story.…')` call in that file. The `周爸的故事` text the customer sees today comes from a `content_blocks` row, not from `story.*` i18n. This is the ground truth that makes `story.*` deletion safe.
- **Editor display is a side-effect, not a contract.** The content editor at `/dashboard/content` builds its tab list from `site_content` rows. Deleting orphan rows changes the editor's visible tabs — that *is* the desired behaviour. No code change to the editor is required.
- **Audit memo for future tickets.** A maintenance script that reports `site_content` rows whose top-level segment is missing from the JSON defaults would have caught the `story.*` orphans automatically. Out of scope for v1, captured here as a follow-up signal.
- **`spec.daily_limit` was added recently (FEAT-13) and is in the JSON.** It's NOT an orphan even though it looks similar — it's actively rendered by the product detail page.
